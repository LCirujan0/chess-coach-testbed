## 1 · Executive Summary

**Overall health: B−.** Engineering discipline is unusually strong for a solo no-build project — documented decisions (`docs/learnings.md`), enforced conventions (constants in config, one state object, no-spoiler rule), a real Playwright/CI safety net, and a maturing design system with token discipline. The grade is held back by three things: the app is still hardwired to one user (hardcoded username `js/puzzle/config.js:124`, "Good morning, Jorge" `today.html:506`, `/` redirecting to the wrong home), the API proxies are open to anyone who finds the URL (`api/coach.js` forwards arbitrary Anthropic bodies with no rate limit or size cap), and the data layer's stated abstraction is fiction — 6+ HTML pages and 5 modules write `localStorage` directly, with silent catches and no quota handling on stores that grow without bound.

**Top 3 risks:** (1) unmetered `/api/coach` spend if the URL leaks — the Anthropic spend cap is the only real guard; (2) unbounded localStorage growth (`game-moves`/`mistakes`) hitting the ~5–10 MB quota and failing silently; (3) zero automated coverage of the core business logic (grading, streak, SRS) — only page-level smoke.

**Top 3 opportunities:** (1) cross-device persistence — **built this pass**: the retention loop now survives a device switch; (2) the new-user funnel — fixing the `/`→`puzzle.html` mis-redirect and first-run path is a one-line + small-copy change with outsized retention impact; (3) surfacing the (already excellent) gamification layer beyond `today.html` — streak/goal/milestones are invisible exactly where the user spends their time (puzzle, session).

## 2 · Repo Map

**Purpose:** chess coaching PWA for players climbing ~950→1500 Elo. Fetches Chess.com games, finds real mistakes with Stockfish, drills them as puzzles with a no-spoiler AI coach. Strategic spine: daily habit / retention.

**Stack:** vanilla HTML/CSS/JS, zero build step, zero runtime npm deps. Vercel static + 2 serverless functions (`api/coach.js`, `api/tag.js`). Stockfish 17.1 WASM bundled in `engine/`; chess.js from esm.sh CDN; Anthropic via the coach proxy. ## 3 · Audit Report

Severity: **C**ritical / **H**igh / **M**edium / **L**ow. Each item marked **[fact]** (verified in source) or **[judgment]**.

### 3.1 Engineering

**Security**

1. **H [fact] — `/api/coach` is an open, unmetered Anthropic proxy.** `api/coach.js:32-63` forwards the request body **as-is** (any model, any `max_tokens`, any size) with no rate limit, origin check, or body cap. The file itself documents the posture (lines 7-20): URL obscurity + Anthropic spend cap. Consequence: anyone who finds the URL can burn the monthly cap in minutes. Acceptable *only* while the URL is unshared; a hard launch blocker. `api/tag.js` is slightly better (20-puzzle cap, `api/tag.js:40-42`) but equally unmetered.
2. **M [fact] — real GitHub PAT in plaintext at `../deploy-token.md:15`.** Context matters: the file documents itself as a deliberate, gitignored, outside-the-repo, 90-day-expiry convenience tradeoff, and the parent folder has no git remote. So: not the "Critical leak" it first appears, but any process/session with folder read access can push to the public prod repo. Recommend: confirm `.gitignore` coverage, keep the expiry rotation, consider a credential manager.
3. **M [fact] — chess.js from esm.sh at runtime, no fallback.** `js/puzzle/lib.js:5`, `js/games/lib.js:2`, `js/board-vision/tracker.js:11`, `js/openings/boot.js:19`. If esm.sh is down or tampered, every board page dies before the global error handler can help (import failure at module parse). SRI isn't available for cross-origin module imports — the honest mitigations are vendoring the file locally (still no build step — it's one static file) or accepting the risk.
4. **L [fact] — `CHESS_COM_USERNAME = 'LCirujano'` hardcoded** (`js/puzzle/config.js:124`). Public info, near-zero security risk; it's a *product* problem (single-user hardwiring), partially superseded by the v0.78 username key.
5. **L [fact] — no CSP / X-Frame-Options / nosniff headers** in `vercel.json`. Cheap to add; matters at launch, not today.

**Architecture & code quality**

6. **H [fact] — the storage abstraction is documented but not real.** CLAUDE.md: "All localStorage reads and writes go through [storage.js]". Reality: `today.html:218-228` declares its own key constants and reads/writes directly; same for `session.html:122-131`, `js/games/boot.js:27-48` (raw key strings, not even in `js/games/config.js`), `js/streak.js:208-215`, `js/puzzle/playout.js`, `js/openings/boot.js`. Consequence: schema changes have no single choke point, and the convention erodes further with every page. (This directly shaped the v0.78 sync design — see §6.)
7. **M [fact] — two redundant motif-classification paths**, already flagged in CLAUDE.md: `js/games/classify.js` (per-mistake call to `/api/coach`, Sonnet, prompt at lines 17-32) vs `js/tagger.js` + `api/tag.js` (batched 20, Haiku, prompt at `api/tag.js:18-24`). Two prompts on the same 17-tag vocabulary that can drift, double Anthropic spend at ingest, duplicated MultiPV parsing. Consolidate on the `/api/tag` path.
8. **M [fact] — `today.html` is a 692-line page with a ~550-line inline IIFE** (`today.html:192-747`) doing session assembly, streak resolution, goal logic, milestones, rendering. It's well-commented but untestable, unsharable, and the page's synchronous-read-at-parse pattern is what forced the sync layer's reload-once design.
9. **M [fact] — silent `catch {}` on nearly every persistence call** (`js/puzzle/storage.js:29-39`, `js/games/boot.js:26-48`). A `QuotaExceededError` is indistinguishable from success; the user loses progress with no signal.
10. **L [fact] — stale docs:** CLAUDE.md's `index.html` description (see §2); "19 ES modules" in `js/puzzle/` is 20; `docs/qa-checklist.md` reconciliation stamp is v0.58 vs app v0.77.

**Performance & data**

11. **H [fact/judgment] — unbounded localStorage growth, no quota handling, no eviction.** `chess-coach-game-moves-v1` stores full SAN move lists per game (`js/games/boot.js:32`), `-mistakes-v1` stores engine lines per puzzle, `-game-meta-v1`/`-scorecards-v1`/`-rating-history-v1` grow per game. At a few KB per game, a few hundred ingested games approaches the ~5 MB browser quota; the failure mode is #9's silent catch. [Judgment: timescale is months-to-years for one user, but the failure is silent data loss.]
12. **M [fact] — eager Stockfish init on every puzzle-page load** (`js/puzzle/boot.js` boots engine before interaction; the deferred Spec 15 Fix 2 is on record in learnings v0.71 as needing its own QA pass). Costs time-to-interactive on slow networks.
13. **L [fact] — caching is actually in good shape**: immutable `/engine`, `/piece`, `/brand-icons`; 24h+SWR on `/data`; no-store HTML (`vercel.json:5-35`). The v0.71/0.72 perf passes did the right things.

**Testing**

14. **H [fact] — the QA pyramid is inverted: good page-level smoke, zero unit coverage of the brain.** Suites a–e cover shell/nav, console-clean loads, board render. Nothing exercises `grade.js` (cp-loss/tiers — the calibrated core), `streak.js`, `review-srs.js`, `coach-stats.js`, or storage merging. The `qa/scripts/` harnesses exist but are run-on-demand, not CI. Parked `fixme` tests (R1.2/R2/R3) are tracked honestly — good practice — but several have been parked across many releases.

**Strengths (engineering)**
- **Decision log discipline** (`docs/learnings.md`) is genuinely exceptional — every reversal records its *why*. This audit was possible largely because of it.
- Conventions that hold: constants in config, one `state` object, global error handlers first (`boot.js`), calibrated thresholds protected by rule.
- The smoke suite ("the test that would have caught the whole firefight") + integrity check + CI on previews is the right minimal safety net.
- No-build architecture is a fit for the project's scale and is honestly documented with its constraint set.
- Data integrity culture: Stockfish-verified openings harness, mechanical line-count-proven refactors (Spec 10), idempotent merges at ingest.

### 3.2 Design / UX

15. **C [fact] — the front door is wrong.** `vercel.json:3` sends `/` to `puzzle.html`; the documented home is `today.html` (CLAUDE.md pages table), and `index.html:5` agrees (`url=/today.html`). A new user lands on a training screen with a default mate-in-1 and no context; the session home, value prop, and "Connect Chess.com" CTA (which is *good* — `today.html:650-656`) are never seen unless they explore the nav. Highest-leverage single-line fix in the repo.
16. **H [fact] — single-user hardwiring is user-visible.** "Good morning, Jorge" (`today.html:504-507`) greets every visitor as Jorge. With v0.78's username key there is now a real identity to greet instead.
17. **H [fact] — gamification is siloed on Today.** Streak chip, freeze, goal ring, milestones, coach's read all render only in `today.html` (`:513`, `:544-575`, `:447-466`). During the actual training (puzzle.html, session.html) none of it is visible; `completed.html` has no celebration moment. The retention mechanics are well-designed (honesty rules, no streak-terror) but under-shown at the moments of effort. [fact for placement; judgment for impact]
18. **H [fact] — the board is inaccessible.** Squares render as unlabeled divs (`js/puzzle/board.js`, `js/board-static.js` — no `aria-label`, no roles, no keyboard path). For the core interaction surface, screen-reader and keyboard users are fully excluded. (Scope honestly: a full a11y pass is large; square labels + focus order is the impactful subset.)
19. **M [fact] — ingest feedback is text-only** (`js/games/boot.js:62-72` `setProgress` strings, no visual bar) for a 10-30s blocking operation that the user must keep open (`beforeunload` warning added in v0.75 — good). A progress bar + per-game ticks would reduce abandonment.
20. **M [judgment, computed] — contrast: `--muted #6E727B` on `--surface2 #F2F4F7` ≈ 4.0:1 and `--accent #2F9E76` on white ≈ 3.3:1 — both below WCAG AA for normal text** (accent is used for the eyebrow/wordmark at small sizes). On plain white, muted ≈ 4.6:1 passes. Worth one token nudge.
21. **M [fact] — known, documented visual debt** (design-system.md backlog #2-4): bespoke cards (`.session`/`.mode-card`/`.loop`/`.op-panel`/`.summary-bar`) vs the canonical `.panel`; `games.html` re-rolls its own button system and doesn't link `train.css` (`games.html:77-99`); ~11-17 hardcoded hexes remain (e.g. `today.html:40` `#CDE9DC`, `insights.html:46` `#FBF3E6`, `session.html` gradient) despite the tokens existing since v0.72.
22. **L [fact] — empty states are mostly good** (Today's "Let's pull your games first" is exemplary; v0.75 fixed Insights' empty state), but `puzzle.html` with zero puzzles shows "No puzzles in this filter." with no CTA to sync.

**Strengths (design/UX)**
- The design system is real and enforced: tokens, single-sourced chrome (`shell.css`), branded header app-wide, typography trio, canonical link order — and the backlog of remaining drift is *written down*.
- Retention layer design quality is high: honest streak with freezes/rest days (`js/streak.js` — pure, testable, supportive copy), data-grounded coach's read with anti-fake-praise guards, SRS derived from existing data with no new key.
- Mobile is genuinely solid: 880px breakpoint, tab-bar/pinned-nav split, no horizontal overflow at 375px (QA-asserted), ~44px buttons.

## 4 · Improvement Strategy

The findings cluster into four themes. The lens for prioritisation: **this app lives or dies on daily habit — anything that breaks learning continuity, trust, or the loop outranks polish.**

**Theme 1 — From Jorge's tool to a product with a user (Critical→High).**
Findings 4, 15, 16, plus the pre-v0.78 device-lock. Target state: one identity (the Chess.com username, now stored in `chess-coach-username-v1`), one front door (`/` → `today.html`), no hardcoded persona anywhere; state follows the user (✅ shipped v0.78). Principle: *the retention loop must survive contact with reality — new device, new user, new day.*

**Theme 2 — Trust the data layer (High).**
Findings 6, 9, 11. Target state: writes fail loudly (quota detection + a visible "storage almost full" state), big blobs have an eviction story (cap `game-moves` to the most recent N games), and new code has one blessed write path (don't boil the ocean migrating old pages — stop the bleeding). Principle: *silent data loss is the fastest way to kill trust in a habit product.*

**Theme 3 — Cost & abuse surface (High, gated on launch).**
Findings 1, 5. Target state: `/api/coach` validates shape (model allowlist, max_tokens cap, body size cap) and rate-limits per IP; security headers on. Principle: *the spend cap is a parachute, not a seatbelt.* Do the cheap body/model validation now; full rate limiting can wait for the launch gate.

**Theme 4 — Show the loop where the work happens (Medium, retention-positive).**
Finding 17, 19, 22. Target state: streak/goal visible in session and on completion; celebration on `completed.html`; sync/ingest progress feels alive. Principle: *reward at the moment of effort, not on the next visit.*

**Deliberately NOT fixing, and why:**
- **No-build / no-framework** — documented, fit for scale; nothing found justifies revisiting.
- **CDN chess.js → vendor it** is a should, not a must (one static file, no build step needed) — scheduled M3, accepted risk meanwhile.
- **Full WCAG pass, visual-regression suite, TypeScript, state-management refactor** — over-engineering at this maturity. Board square labels only (the impactful a11y subset).
- **Migrating every inline page script to modules** — only `today.html` earns it (it's the home and the most-edited file); the others are stable.
- **`anya_state` table / parent-repo hygiene** — outside this app's scope; noted only.

**"Done" signals:** a fresh browser reaching `/` sees Today and a username prompt; device B shows device A's streak (✅ verified); a 413/429 from `/api/coach` is observable in Vercel logs under abuse; `npm test` exercises grade/streak/SRS/merge logic in node; quota at 80% surfaces a user-visible warning; streak visible on session + completed.

### What shipped

| Piece | Where |
|---|---|

| Constants (URL, publishable key, `STORAGE_KEY_USERNAME`, `SYNC_KEYS`, debounce) | `js/puzzle/config.js` (per project rule 2) |

| Decision log | `docs/learnings.md` v0.78; CLAUDE.md updated |

### Schema (and why per-key rows, not one blob)

```sql
create table public.knightpath_state (
  username   text not null check (username = lower(username) and length(username) between 1 and 64),
  key        text not null check (key like 'chess-coach-%' and length(key) <= 80),
  value      jsonb,
  updated_at timestamptz not null default now(),   -- trigger-maintained; client clocks untrusted
  primary key (username, key)
);
-- RLS: anon may select/insert/update — never delete.
```

The access pattern is "read everything on boot, write what changed on events". A single JSON blob makes every push re-send the whole state (the mistakes store alone can reach ~1 MB); per-key rows keep event pushes to just the dirty keys (one upsert with `Prefer: resolution=merge-duplicates`), give a server-side per-key `updated_at`, and boot is still **one GET**. It's the blob model's read simplicity with row-level write granularity — the right hybrid for this shape.

### Identity

Chess.com username, lowercased, stored in `chess-coach-username-v1`. First load without one shows a **non-blocking, token-styled banner** ("What's your Chess.com username? … Not now"); until saved, the app sends **nothing** (which also keeps QA runs fully offline). "Not now" defers for the tab session. Subsequent loads skip the prompt.

### Sync strategy

- **On load:** pull all rows for the user → merge per-key → write merged values to localStorage → push any key where local won → if the merge changed local state, **reload once** (sessionStorage-guarded, 15s anti-loop) so the synchronous inline page scripts re-read the stores. Subsequent opens don't reload.
- **On meaningful events:** a `Storage.prototype.setItem` hook schedules a **debounced (2.5s) push of changed keys only**. Puzzle resolved → attempts write → push; session completed → complete-flag + streak writes → one coalesced push. Filter prefs and other non-synced keys never trigger network.
- **On pagehide/hidden:** keepalive flush (skipped >60 KB — the next load's pull/merge/push covers it).
- **Logged deviation from the brief:** the brief said to wrap `js/puzzle/storage.js`. In reality the events this feature exists for happen *outside* it — `streak.js:212`, `session.html:131`, `playout.js`, `openings/boot.js`, the inline pages — so wrapping the puzzle module alone would miss the streak itself. The prototype hook catches every write with zero call-site changes. (Also learned the hard way and fixed: patching `localStorage.setItem` on the *instance* silently no-ops — Storage objects intercept property sets — so the hook lives on `Storage.prototype`.)

### Synced keys & conflict rules

All 9 priority keys from the brief, plus three cheap gamification keys device B needs for a coherent Today (`session-complete-v1`, `daily-goal-v1`, `mastery-seen-v1`):

| Key | Merge rule |
|---|---|
| `streak-v1` | higher `current` wins; `longest`/freezes = max; day-lists union |
| `attempts-v1` | union of ids; per id, later `lastAt` wins |
| `mistakes-v1` | union by id |
